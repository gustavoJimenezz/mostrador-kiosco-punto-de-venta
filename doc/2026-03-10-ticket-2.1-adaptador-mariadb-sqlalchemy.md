# Adaptador MariaDB con SQLAlchemy 2.0

**Fecha:** 2026-03-10
**Ticket:** 2.1 — Epic 2: Infraestructura y Adaptadores de Datos

## Resumen

Implementación de la capa de infraestructura de persistencia usando SQLAlchemy 2.0 con mapeo imperativo (classical mapping). El objetivo es conectar las entidades de dominio puras con MariaDB sin que el dominio importe nada de SQLAlchemy, respetando la regla fundamental de la arquitectura hexagonal.

## Cambios Principales

- Creada la estructura `src/infrastructure/persistence/` con 4 módulos
- Definido el esquema completo de 5 tablas en SQLAlchemy Core (`tables.py`)
- Implementado mapeo imperativo de `Product` y `CashClose` (`mappings.py`)
- Implementado `MariadbProductRepository` como adaptador del puerto `ProductRepository`
- Configurado engine con pool de conexiones optimizado para kioscos (`database.py`)
- Inicializado Alembic con migración inicial que crea todas las tablas en MariaDB InnoDB
- Agregado `PyMySQL >= 1.1.0` como driver puro Python (sin dependencias C)

## Flujo de Trabajo

```
[main.py]
   │
   ├── create_mariadb_engine(url)      → Engine con QueuePool configurado
   ├── configure_mappings()            → Product y CashClose instrumentados por SQLAlchemy
   └── create_session_factory(engine)  → SessionFactory lista para usar

[Caso de Uso]
   │
   ├── with SessionFactory() as session:
   │       repo = MariadbProductRepository(session)
   │       product = repo.get_by_barcode("7790895000115")
   │       session.commit()
   │
   └── [repo] → [SQLAlchemy ORM] → [PyMySQL] → [MariaDB]
```

**Punto de entrada:** `main.py` (Composition Root) llama a `configure_mappings()` una sola vez al arrancar.

**Proceso:** Los casos de uso crean una sesión, instancian el repositorio y delegan la persistencia. El repositorio emite `flush()` (no `commit`) para obtener IDs sin cerrar la transacción.

**Resultado:** Las entidades de dominio se persisten y recuperan sin conocer la existencia de SQLAlchemy ni MariaDB.

## Notas Técnicas

### Decisiones de diseño

**`Sale.total_amount` no se mapea como atributo ORM:** Es un `@property` calculado en el dominio a partir de `sale_items`. La columna existe en la tabla para consultas analíticas (suma del día), pero el repositorio la materializa al hacer `INSERT` con el valor calculado. Esto evita conflicto entre el descriptor Python y el `InstrumentedAttribute` de SQLAlchemy.

**`SaleItem` y `Sale` no tienen mapeo ORM completo:** `SaleItem` no expone `sale_id` como atributo de dominio (accede a los ítems siempre a través de `Sale.items`). La persistencia atómica de ventas se implementará en el caso de uso `ProcessSale` (Ticket futuro) usando Core SQL, lo que garantiza la transacción completa (stock + venta + caja) en un solo commit.

**`configure_mappings()` es idempotente:** Usa un flag `_mappings_configured` para evitar el error `InvalidRequestError: already mapped` en entornos de test donde múltiples módulos podrían llamarla.

**FullText index via DDL explícito:** SQLAlchemy Core no abstrae `CREATE FULLTEXT INDEX`. Se agrega en la migración con `op.execute()`. En producción permite `MATCH(name) AGAINST(?)` para búsquedas fuzzy < 50ms con 5.000 registros (Ticket 2.2).

### Pool de conexiones

| Parámetro | Valor | Justificación |
|-----------|-------|---------------|
| `pool_size` | 5 | Suficiente para una terminal POS single-caja |
| `pool_pre_ping` | `True` | Previene "MySQL server has gone away" en kioscos inactivos |
| `pool_recycle` | 3600 | Renueva conexiones cada hora (cortes de luz frecuentes en verano AR) |
| `pool_timeout` | 30 | Falla rápido si la DB no responde en 30s |
