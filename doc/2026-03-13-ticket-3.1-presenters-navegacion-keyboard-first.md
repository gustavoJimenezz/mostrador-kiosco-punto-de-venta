# Presenters y Navegación Keyboard-First

**Fecha:** 2026-03-13
**Ticket:** 3.1 — Epic 3: Interfaz de Usuario y Adaptador Primario

## Resumen

Implementación de la capa de presentación usando el patrón **MVP (Model-View-Presenter)**. Se introduce `SalePresenter` como clase Python pura (sin dependencias de PySide6), `MainWindow` como adaptador Qt que carga un archivo `.ui` dinámicamente, y tres workers `QThread` para ejecutar operaciones de DB en hilos separados. El hilo principal de Qt nunca se bloquea, garantizando que el lector de barras no pierda caracteres durante búsquedas.

## Cambios Principales

- Creada capa `src/application/use_cases/` con `GetProductByCode` y `ProcessSale`
- Definido puerto `SaleRepository` (Protocol) en `src/domain/ports/`
- Implementado `MariadbSaleRepository`: persiste sale + sale_items + decrementa stock en una sola transacción atómica
- Implementado `SalePresenter` con estado de carrito, cálculo de totales y coordinación de flujos
- Definido protocolo `ISaleView` para desacoplar el presenter de Qt
- Creados workers `SearchByBarcodeWorker`, `SearchByNameWorker`, `ProcessSaleWorker` (cada uno con su propia sesión SQLAlchemy)
- Implementada `MainWindow` con carga dinámica de `main_window.ui` y atajos F1/F2/F4/F10/Esc
- Creado `src/main.py` como Composition Root con inyección de dependencias
- 29 tests unitarios del presenter sin necesidad de Qt instalado (238 total, 0 fallos)

## Flujo de Trabajo

### Escaneo de barcode

```
[Cajero escanea EAN-13]
        │
        ▼
[barcode_input.returnPressed]  ← hilo principal Qt
        │
        ▼
[MainWindow._on_barcode_entered]
        │  crea worker, conecta señales
        ▼
[SearchByBarcodeWorker.run()]   ← hilo separado
  GetProductByCode.execute()
  MariadbProductRepository.get_by_barcode()
        │
        ├── encontrado ──► Signal product_found(Product)
        │                         │
        └── no encontrado ──► Signal not_found(barcode)
                                  │
        ┌─────────────────────────┘
        ▼
[presenter.on_barcode_found(product)]  ← hilo principal (entrega via signal)
  _add_product_to_cart(product)
  view.show_product_in_cart(product, qty)
  view.update_total(total)
```

### Confirmación de venta (F4)

```
[Cajero presiona F4]
        │
        ▼
[presenter.on_confirm_sale_requested()]
        │  carrito vacío → view.show_error()
        │  carrito con ítems → view.show_payment_dialog()
        ▼
[_PaymentDialog modal]   ← teclado: flechas + Enter
        │
        ▼
[MainWindow crea ProcessSaleWorker]  ← hilo separado
  ProcessSale.execute(cart, payment_method)
    ↳ valida stock de cada ítem
    ↳ construye Sale + SaleItems (price_at_sale = snapshot inmutable)
    ↳ MariadbSaleRepository.save(sale)
        INSERT sales + INSERT sale_items + UPDATE products.stock
        COMMIT (o ROLLBACK si falla)
        │
        ▼
[presenter.on_sale_completed(sale)]
  view.show_sale_confirmed(sale)
  _clear_cart()  ←  pantalla lista para la próxima venta
```

**Punto de entrada:** `barcode_input.returnPressed` o atajos de teclado F1-F10.

**Proceso:** MainWindow delega toda la lógica al presenter. Workers QThread resuelven la DB en segundo plano y emiten señales al hilo principal. El presenter nunca importa PySide6.

**Resultado:** Venta persistida en MariaDB con `price_at_sale` inmutable, stock decrementado y carrito limpio para la próxima operación.

## Notas Técnicas

### Patrón MVP aplicado

El presenter es Python puro: no importa `PySide6`. Esto permite ejecutar los 29 tests del presenter en Linux sin Qt instalado:

```
MainWindow (View)          SalePresenter            Workers (QThread)
─────────────────          ─────────────            ─────────────────
Registra atajos F1-F10  →  on_barcode_found()    ←  SearchByBarcodeWorker
Implementa ISaleView    ←  view.update_total()      SearchByNameWorker
Crea workers            →  on_confirm_sale_req()    ProcessSaleWorker
Conecta señales         →  get_cart()
```

### Sesiones SQLAlchemy por hilo

Cada worker recibe `session_factory` (callable) en lugar de una `Session` ya creada. Compartir sesiones entre hilos en SQLAlchemy no es thread-safe:

```python
# Correcto: cada worker crea su propia sesión
worker = SearchByBarcodeWorker(session_factory, barcode)

# Incorrecto: sesión compartida entre hilos
worker = SearchByBarcodeWorker(existing_session, barcode)
```

### Atomicidad en ProcessSale

`MariadbSaleRepository.save()` garantiza la regla de negocio crítica:

```
INSERT sales
INSERT sale_items  (price_at_sale = snapshot inmutable del precio actual)
UPDATE products.stock - quantity
─────────────────────────────────
COMMIT  →  éxito
ROLLBACK →  cualquier fallo (stock insuficiente, FK violation, etc.)
```
