**Entorno:** Ubuntu / Linux

## Resumen

Definición de los puertos de salida (output ports) del dominio mediante `Protocol` con `runtime_checkable`. Se formalizaron las interfaces `ProductRepository` y `PrinterBase` que toda implementación de infraestructura debe cumplir, garantizando que el core del dominio permanezca libre de dependencias externas. Se extrajo el `InMemoryProductRepository` a un módulo de mocks compartido y se amplió la suite de tests con cobertura del contrato completo incluyendo el método `delete` y la interfaz de impresión.

**DoD alcanzado:** Todos los criterios de aceptación del issue #2 completados.

## Flujo de Trabajo

### Inyección de dependencias vía puertos

```
[Caso de Uso] → [ProductRepository (Protocol)] ← [InMemoryProductRepository (tests)]
                                                ← [SQLAlchemyProductRepository (Epic 2)]
```

1. Los casos de uso solo conocen el `Protocol`; nunca la implementación concreta
2. En tests se inyecta `InMemoryProductRepository` (sin DB, sin IO)
3. En producción se inyectará `SQLAlchemyProductRepository` (Epic 2)
4. El `main.py` (Composition Root) es el único lugar donde se instancia el adaptador real

### Verificación del contrato en runtime

```
[InMemoryProductRepository()] → [isinstance(repo, ProductRepository)] → [True]
```

1. `@runtime_checkable` permite verificar el contrato en tiempo de ejecución
2. Si un adaptador no implementa todos los métodos del Protocol, `isinstance` retorna `False`
3. Los tests de contrato detectan regresiones si algún método se omite o se renombra

### Flujo de delete en el repositorio en memoria

```
[repo.delete(id)] → [_store.pop(id, None)] → [silencioso si no existe]
```

1. La operación es idempotente: no lanza excepción si el ID no existe
2. Reduce `list_all()` en uno tras la eliminación
3. `get_by_id(id)` retorna `None` inmediatamente después del delete

## Notas Técnicas

### Por qué `Protocol` y no `ABC`

Se mantiene `Protocol` (structural subtyping) para ambas interfaces. Esto permite que cualquier clase que implemente los métodos correctos satisfaga el contrato sin herencia explícita, lo que facilita la integración con librerías de terceros y simplifica los mocks en tests.

### Convención de delete silencioso

`delete(product_id)` no lanza excepción si el ID no existe. Esto es consistente con la semántica de operaciones idempotentes en sistemas de persistencia: el estado deseado (que el producto no exista) ya se cumple.

### PrinterBase: interfaz mínima para Epic 4

`PrinterBase` solo expone `print_ticket(sale: Sale) -> None` en este ticket. La implementación ESC/POS con pywin32 y rutas UNC (`\\localhost\Ticketera`) se desarrolla en el Epic 4 (entorno Windows 11). La interfaz permite testear los casos de uso de venta sin hardware real usando `MockPrinter`.

### Compatibilidad hacia atrás

`repositories.py` re-exporta `ProductRepository` desde `product_repository.py`. Los imports existentes en `test_repositories_port.py` no requirieron cambio de ruta.
