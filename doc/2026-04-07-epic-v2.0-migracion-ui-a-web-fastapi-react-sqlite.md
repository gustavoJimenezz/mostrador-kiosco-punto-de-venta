### ✅ Ticket 1.1 — `database.py`: agregar `create_sqlite_engine()` + adaptar `alembic/env.py` para SQLite

Se agrega soporte completo de SQLite como motor de base de datos local para el kiosco,
en reemplazo de MariaDB local. Esto elimina la necesidad de instalar MariaDB en la PC
del kiosco y permite que la app funcione con un archivo `.db` portable.

**Cambios:**
- Nuevo `create_sqlite_engine()` en `database.py` con `StaticPool`, WAL y foreign keys activados via `event.listens_for`
- `alembic/env.py` adaptado para emitir `render_as_batch=True` cuando el dialecto es SQLite (necesario para emular `ALTER TABLE`)
- Migración `1bafd69c0714`: `CREATE FULLTEXT INDEX` ahora es condicional — se omite en SQLite
- Migración `f7c3b2a1d9e8`: `INSERT IGNORE` adaptado a `INSERT OR IGNORE` en SQLite

**Flujo:**
```
DATABASE_URL=sqlite:///~/.local/share/kiosco-pos/pos.db
        │
        ▼
create_sqlite_engine(db_path)
        ├─ StaticPool (una sola conexión, un solo worker uvicorn)
        ├─ check_same_thread=False
        ├─ PRAGMA journal_mode=WAL   ← se activa en cada nueva conexión
        └─ PRAGMA foreign_keys=ON   ← se activa en cada nueva conexión
        │
        ▼
alembic upgrade head
        ├─ render_as_batch=True          ← env.py detecta dialect SQLite
        ├─ SKIP CREATE FULLTEXT INDEX    ← migración 1bafd69c0714
        └─ INSERT OR IGNORE              ← migración f7c3b2a1d9e8
```

### ✅ Ticket 1.2 — `sqlite_product_repository.py`: sobreescribir búsqueda FULL TEXT → ILIKE

Se crea `SqliteProductRepository` que hereda de `MariadbProductRepository` y sobreescribe
únicamente el método `_search_by_name_fulltext`, reemplazando `MATCH...AGAINST` (FullText
de MariaDB, no disponible en SQLite) por `ILIKE`. El resto de la lógica es idéntica.

**Flujo:**
```
search_by_name(query)
        │
        ├─ len(query) < 3 → _search_by_name_ilike()
        └─ len(query) >= 3 → _search_by_name_fulltext()  [sobreescrito]
                                      └─ _search_by_name_ilike()  ← SQLite fallback
```

## Notas Técnicas

**`StaticPool`:** uvicorn corre con `--workers 1`. Una sola conexión SQLite persistente es
suficiente. Si en el futuro se aumentan workers, cambiar a `NullPool` con WAL habilitado.

**`render_as_batch`:** SQLite no soporta `ALTER TABLE` nativo. Alembic emula estas
operaciones creando una tabla temporal, copiando datos y renombrando. Transparente para el
desarrollador.

**Búsqueda ILIKE vs FullText:** para ~5,000 SKUs en el kiosco, ILIKE es suficiente.
El FullText index de MariaDB se mantiene intacto para el servidor remoto.

**Sin cambios en MariaDB:** `MariadbProductRepository` y todas las migraciones siguen
funcionando sin cambios. 559 tests existentes pasan sin modificación.

### ✅ Ticket 1.4 — Routers FastAPI + `app.py` + `dependencies.py`

Se crea la capa web completa en `src/infrastructure/web/`.

**`app.py` — application factory:**
- Lifespan: detecta SQLite vs MariaDB por `DATABASE_URL`, crea engine, ejecuta `alembic upgrade head`, llama `configure_mappings()`
- SessionMiddleware para autenticación por cookie firmada (`itsdangerous`)
- CORS habilitado solo para `localhost:5173` (Vite dev server)
- Monta `frontend/dist/` como archivos estáticos si existe el bundle

**`dependencies.py` — inyección de dependencias:**
- `get_session(request)` → sesión SQLAlchemy por request, con rollback automático
- `require_auth(request)` → verifica cookie de sesión, retorna 401 si no hay sesión
- `require_admin(auth)` → verifica rol `admin`, retorna 403 si no es admin
- Factories para cada repositorio (`get_product_repo`, `get_cash_repo`, etc.)

**Routers implementados:**

| Router | Endpoints |
|--------|-----------|
| `auth.py` | `GET /api/auth/users`, `POST /api/auth/login`, `POST /api/auth/logout`, `GET /api/auth/me` |
| `pos.py` | `GET /api/products/barcode/{bc}`, `GET /api/products/search?q=`, `POST /api/sales` |
| `cash.py` | `GET /api/cash/state`, `POST /api/cash/open`, `POST /api/cash/close`, `POST /api/cash/movements` |
| `admin.py` | CRUD `/api/products`, `/api/categories`, `GET /api/users` |
| `import_.py` | `POST /api/import`, `GET /api/import/status` |

**Regla crítica en `POST /api/sales`:** el endpoint recibe `{items:[{product_id, qty}]}`. Recarga
cada `Product` desde DB y asigna `price_at_sale = product.current_price.amount` server-side.
El cliente nunca envía precios.

**Flujo de venta:**
```
POST /api/sales  {items:[{product_id, qty}], payment_method}
        │
        ├─ Cargar Product desde DB por product_id (precio server-side)
        ├─ ProcessSale.execute(cart, payment_method)
        ├─ session.commit()  ← atómico: sale + sale_items + stock
        └─ SaleResponse {id, total_amount (str), items}
```

**Importación en background:**
```
POST /api/import  (multipart)
        ├─ Guardar archivo en /tmp
        ├─ threading.Thread(target=_run_import)
        └─ HTTP 202 Accepted (no bloquea)

GET /api/import/status
        └─ {state: "running"|"done"|"error", inserted, updated, skipped}
```

### ✅ Ticket 1.6 — Sync EOD: `eod_sync.py`

Sincronización End-of-Day del arqueo de caja local (SQLite) al servidor remoto (MariaDB).
Se dispara automáticamente en background al ejecutar `POST /api/cash/close` si la variable
`REMOTE_DATABASE_URL` está configurada.

**Flujo:**
```
POST /api/cash/close
        │
        ├─ CloseCashClose.execute()
        ├─ session.commit()
        └─ _trigger_eod_sync_if_configured(cash_close_id)
                └─ threading.Thread → EodSync.sync(cash_close_id)
                        ├─ _fetch_local_sales()   ← SQLite local
                        ├─ _upsert_remote_sales() ← MariaDB remoto (INSERT IGNORE por UUID)
                        └─ _log_sync()            ← tabla sync_log en SQLite
```

**Idempotencia:** cada venta tiene UUID. Si ya existe en el remoto, se omite sin error.
Protege contra dobles envíos por reintentos.

**Sin servidor remoto:** si `REMOTE_DATABASE_URL` no está definida, el sync se omite
silenciosamente. El cierre de caja local igual se completa.

## Ticket 1.10 — Instrucciones de prueba en hardware E5700 + Ubuntu 22.04

### Prerequisitos en la PC de build (Ubuntu 22.04, x86_64)

```bash
sudo apt install python3-dev python3-venv build-essential dpkg-dev nodejs npm
curl -sSL https://install.python-poetry.org | python3 -
```

Node >= 18 requerido para el build React. Verificar:

```bash
node --version   # >= 18.x
npm --version
```

### Paso 1 — Generar el paquete .deb

Desde la raíz del repositorio en la PC de build:

```bash
bash packaging/build_deb.sh
```

Genera: `dist/kiosco-pos_0.1.0_amd64.deb`

El script realiza:
1. `npm ci && npm run build` en `frontend/` → produce `frontend/dist/`
2. Crea un virtualenv Python en `/tmp/build_deb_*/usr/lib/kiosco-pos/venv/`
3. Instala dependencias Python (fastapi, uvicorn, sqlalchemy, alembic, etc.)
4. Copia código fuente, frontend compilado y archivos de packaging
5. Genera `DEBIAN/control`, `postinst` y `prerm`
6. Empaqueta con `dpkg-deb`

```bash
cp dist/kiosco-pos_0.1.0_amd64.deb /media/usb/
```

### Paso 3 — Instalar en el kiosco E5700 (Ubuntu 22.04)

```bash
sudo dpkg -i kiosco-pos_0.1.0_amd64.deb
```

1. Crea usuario del sistema `kiosco` si no existe

Verificar que el servicio arrancó:


Ver logs en tiempo real:


### Problemas conocidos y soluciones

| Problema | Causa probable | Solución |
|----------|---------------|----------|
| `SIGILL` al iniciar uvicorn | CPU no soporta instrucción usada por una dep compilada | Verificar que el venv fue compilado en E5700 o en CPU compatible. El .deb debe compilarse en la misma arquitectura. |
| Chromium abre antes de que el API esté lista | Race condition entre autostart y systemd | Agregar `ExecStartPost=/bin/sleep 3` en el `.service` o delay en el `.desktop` |
| `address already in use :8000` | Puerto ocupado por instancia anterior | `systemctl restart kiosco-pos.service` |
| Base de datos no se crea | Permisos en `~/.local/share/kiosco-pos/` | `sudo -u kiosco mkdir -p /home/kiosco/.local/share/kiosco-pos` |
| Alembic falla al migrar | `alembic.ini` no apunta al módulo correcto | Verificar `script_location` en `alembic.ini` y que `PYTHONPATH` incluye `/usr/lib/kiosco-pos` |
