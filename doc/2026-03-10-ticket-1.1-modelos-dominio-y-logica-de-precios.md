## Flujo de Trabajo

### Cálculo de Precio de Venta

```
[Costo (Decimal)] → [Price(costo)] → [.apply_margin(margen%)] → [Price(precio_venta)]
```

1. El costo del proveedor se envuelve en un `Price` (redondeo automático a 2 decimales)
2. Se aplica el margen con la fórmula: `precio = costo × (1 + margen / 100)`
3. El resultado se redondea con `ROUND_HALF_UP` y se retorna como nuevo `Price`

### Registro de una Venta

```
[SaleItem(product_id, qty, price_at_sale)] → [Sale.items] → [Sale.total_amount]
```

1. Cada `SaleItem` captura el `price_at_sale` en el momento de la venta (inmutable)
2. La `Sale` acumula los subtotales con `Price.__add__`
3. El `total_amount` se calcula dinámicamente sumando todos los ítems

### Arqueo de Caja

```
[CashClose.opening_amount] + [ventas en efectivo] → [expected_cash]
[closing_amount] - [expected_cash] → [cash_difference]
```

1. Al inicio del día se registra el fondo de cambio (`opening_amount`)
2. Cada venta acumula su monto al método de pago correspondiente
3. Al cierre se compara el efectivo contado con el esperado para detectar diferencias

## Notas Técnicas

### Aritmética monetaria
- `Price` usa `@dataclass(frozen=True)` para garantizar inmutabilidad
- Todo valor se convierte a `Decimal` en el constructor y se redondea con `ROUND_HALF_UP` a 2 decimales
- `float` está explícitamente rechazado en el constructor para evitar errores de punto flotante

### Regla de price_at_sale
- `SaleItem` es `frozen=True`: una vez registrado el precio al momento de la venta, no puede modificarse
- Los casos de uso (Epic 2+) son responsables de capturar `product.current_price.amount` antes de crear el `SaleItem`

### Regla de dominio puro
- Ningún archivo en `src/domain/` importa SQLAlchemy, Pydantic, PySide6 ni ninguna librería externa
- Los tests corren sin base de datos usando `InMemoryProductRepository`

### Configuración de cobertura
- `pyproject.toml` define `fail_under=100` sobre `src/domain/`
- Ejecutar con: `.venv/bin/pytest tests/domain/ --cov=src/domain`
